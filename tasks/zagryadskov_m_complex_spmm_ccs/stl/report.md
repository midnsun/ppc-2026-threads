## 1. Контекст

Рассматривается параллельная реализация умножения разреженных матриц в формате **CCS** [1]
с использованием `std::thread` [2]. Данная реализация выполняется в качестве учебной для
проверки способностей ручного распределения данных и функций потокам без использования
примитивов OpenMP и OneTBB.

Цель:

- реализовать параллельную версию при помощи `std::thread`, используя ручной параллелизм
в виде `create -> work -> join`
- исследовать масштабируемость алгоритма

---

## 2. Постановка задачи

Необходимо для двух матриц в формате **CCS** вычислить их произведение и затем сохранить его в формате **CCS**.
Произведение двух матриц A = a[i, j], B = b[j, k] (i = 1, m; j = 1, n; k = 1, p; a[i, j] ∈ **C**; b[i, j] ∈ **C**)
равно матрице C со элементами, выражающимися по формуле: C = c[i, k] = Σ(a[i, j] * b[j, k]), j ∈ 1, n.

### Условия корректности

- Проверяется: **A.n = B.m**
- Формат результата: CCS
- Погрешность: **1e-14**
- Сравнение с SEQ

---

## 3. Базовый алгоритм

Используется тот же двухфазный алгоритм, что и в SEQ:

1. Symbolic phase — для каждого столбца правой матрицы определяется структура соответствующего столбца результата:

- Выполняется обход ненулевых элементов и формируется множество индексов строк
- Вычисляется массив **col_ptr** результирующей матрицы

1. Numeric phase — для каждого столбца правой матрицы вычисляется столбец итоговой матрицы:

- Для каждого столбца используются локальные для потоков вектора **acc** аккумулирования результата и **marker**
- Численный результат записывается в заранее выделенные векторы результирующей матрицы

Параллелизм вводится на уровне обработки столбцов матрицы B. По сравнению с SEQ,
STL-версия меняет только организацию вычислений из-за необходимости разбить диапазон столбцов
на области, обрабатываемые каждым потоком.

---

## 4. Схема распараллеливания

### Область параллелизма

Параллелизуется обе фазы по следующему сценарию:
1. Каждому потоку в зависимости от его номера назначается диапазон строк
2. Этот диапазон записывается как аргумент для функций `SpMMSymbolic` и `SpMMNumeric` и
добавляется в контейнер потоков `threads`
3. В цикле по `threads` для каждого потока вызывается `.join()`

Приведённый ниже листинг подтверждает вышеописанный алгоритм параллелизма:
```cpp
for (int tid = 0; tid < num_threads; ++tid) {
    int jstart = (tid * b.n) / num_threads;
    int jend = ((tid + 1) * b.n) / num_threads;
    threads[tid] = std::thread(SpMMSymbolic, std::cref(a), std::cref(b), std::ref(c.col_ptr), jstart, jend);
  }
  for (auto &th : threads) {
    th.join();
  }
```

### Разделение данных

- `a`, `b`, `c` — общие, передаются по ссылке
- `marker`, `acc`, `rows` — **локальные внутри функций**

При этом локальные результаты хранятся в матрице **C**, но каждый поток записывает в свои
столбцы и гонки данных не происходит

### Синхронизация

- отсутствует (каждый поток пишет в свой диапазон `c`)
- нет `atomic`, `mutex`, `condition_variable`

### Барьеры

- неявный барьер в виде `th.join()` после заполнения контейнера потоков

## 5. Детали реализации

Файлы:

- `seq/include/ops_stl.hpp`
- `seq/src/ops_stl.cpp`

Pipeline реализации:

- **ValidationImpl**
  - проверка условия `A.n == B.m`

- **PreProcessingImpl**
  - входные данные уже готовы

- **RunImpl**
  - создание потоков, назначение им соответствующих функций
  - параллельный вызов:
    - `SpMMSymbolic`
    - `SpMMNumeric`

- **PostProcessingImpl**
  - результат уже сформирован

### Особенности

- Используется двухфазный подход, засчёт которого избегаются гонки данных
- Каждый поток работает со своим диапазоном столбцов **B**

## 6. Проверка корректности

Корректность проверялась:

- сравнением с реализацией SEQ на следующих данных:
  - граничные случаи
  - случайные разреженные матрицы

Сравнение выполняется с точностью **1e-14**.

## Приложение

Фрагмент показывает верхний уровень SpMM, на котором иллюстрируется принцип распределения
данных по потокам. Здесь сперва для каждого потока вычисляются индексы столбцов матрицы B,
затем выполняются фазы symbolic и numeric, каждая вычисляется параллельно.

```cpp
void ZagryadskovMComplexSpMMCCSOMP::SpMM(const CCS &a, const CCS &b, CCS &c) {
  c.m = a.m;
  c.n = b.n;
  const int num_threads = ppc::util::GetNumThreads();

  std::complex<double> zero(0.0, 0.0);
  c.col_ptr.assign(c.n + 1, 0);

#pragma omp parallel default(none) shared(num_threads, a, b, c) num_threads(ppc::util::GetNumThreads())
  {
    int tid = omp_get_thread_num();
    int jstart = (tid * b.n) / num_threads;
    int jend = ((tid + 1) * b.n) / num_threads;
    SpMMSymbolic(a, b, c.col_ptr, jstart, jend);
  }

  for (int j = 0; j < c.n; ++j) {
    c.col_ptr[j + 1] += c.col_ptr[j];
  }
  int nnz = c.col_ptr[b.n];
  c.row_ind.resize(nnz);
  c.values.resize(nnz);
#pragma omp parallel default(none) shared(num_threads, a, b, c, zero) num_threads(ppc::util::GetNumThreads())
  {
    int tid = omp_get_thread_num();
    int jstart = (tid * b.n) / num_threads;
    int jend = ((tid + 1) * b.n) / num_threads;
    SpMMNumeric(a, b, c, zero, jstart, jend);
  }
}
```
