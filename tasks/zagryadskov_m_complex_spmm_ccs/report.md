## 1. Введение

В данной работе исследованы несколько реализаций алгоритма умножения двух разреженных матриц в формате
**CCS** (Compressed Column Storage, см. [1]) с элементами комплексного типа `std::complex<double>`.
Рассматривались реализации:

- Без использования технологий параллелизма (последовательная версия, технология **SEQ**);
- С использованием библиотеки OpenMP [2] (технология **OMP**);
- С использованием библиотеки OneTBB [3] (технология **TBB**);
- С использованием стандартной библиотеки `threads` (технология **STL**);
- С использованием MPI [4] для параллелизма на уровне процессов совместно с OpenMP
для параллелизма на уровне потоков (технология **ALL**)

Данная задача подходит для параллелизма, поскольку она может требовать большого числа вычислений
в сравнении с потребляемой памятью, а для организации параллельных вычислений можно
использовать одинаково вычислительное ядро для всех технологий

## 2. Единая постановка задачи

Необходимо для двух матриц в формате **CCS** вычислить их произведение и затем сохранить его в формате **CCS**.
Произведение двух матриц A = a[i, j], B = b[j, k] (i = 1, m; j = 1, n; k = 1, p; a[i, j] ∈ **C**; b[i, j] ∈ **C**)
равно матрице C со элементами, выражающимися по формуле: C = c[i, k] = Σ(a[i, j] * b[j, k]), j ∈ 1, n.
В формате **CCS** разреженная матрица m x n с nz - количеством ненулевых элементов представляется в виде 3х массивов:

- Массив `row_ind` размера nz, содержащий координату строки соответствующего элемента;
- Массив `values` размера nz, содержащий значение соотвествующего ненулевого элемента;
- Массив `col_ptr` размера n + 1, содержащий индекс начала нового столбца в массивах `row_ind` и `values`;

В ходе выполнения задачи требуется:

1. Реализовать последовательную версию данного алгоритма;
2. Реализовать параллельные версии с использованием технологий OMP, TBB, STL и ALL
3. Проверить корректность реализаций и провести экспериментальные замеры времени выполнения;
4. Сравнить время работы параллельных реализации с использованием различного количества потоков
и процессов (для ALL) с последовательной реализацией

### Условия корректности

- Проверяется: **A.n = B.m**
- Формат результата: CCS
- Погрешность: **1e-14**
- Сравнение с SEQ

## 3. Единая методика эксперимента

### Экспериментальная среда

- CPU: Intel Core 6-core i5-12400F
- RAM: 64 GB  
- OS: Windows 11  
- Компилятор: MSVC  
- Сборка: Release

Переменная среды PPC_NUM_THREADS используется для задания количества потоков
Для задания количества потоков используется через строку запуска: `mpiexec -<количество процессов>`

### Тестовые данные

- несколько матриц небольшой размерности для `functional`-тестов, проверяющих
граничные случаи. Результат сравнивается с результатом, полученным вручную
- `performance`-тест использует сгенерированную ленточную матрицу размера **40000 x 40000**,
ширина ленты 100.

### Используемые метрики

- Time (время) - медиана запусков `task_run`
- Speedup (ускорение) - отношение времени последовательной реализации к времени сравниваемой реализации
- Efficiency (эффективность) = speedup / workers, где workers - количество потоков (для OMP, TBB, STL),
или произведение количества потоков на количество процессов (для ALL)

## 6. Интерпретация результатов

`SEQ` является инвариантом времени для всех задач.
`OMP` показывает неплохой прирост производительности и малые накладные расходы. Однако,
в связи с устаревшей реализацией библиотеки для MSVC результат не такой хороший, как мог быть.
`TBB` также показывает неплохой прирост, однако для этих тестовых данных не требуется сложная балансировка
нагрузки, поэтому `TBB` показывает себя немного хуже `OMP`.
`STL` показывает ускорение, сопоставимое с `OMP`, а иногда даже лучше. Накладные расходы в этой технологии
минимальны. В связи с устаревшей версией OpenMP и отстуствием жестких требований к балансировке, эта технология
показывает очень хороший результат.
`ALL` показывает себя хуже остальных технологий, но все еще является неплохо масштабируемым.
Прирост производительности достаточно существенный, однако технология MPI направлена на системы
с распределенной памятью и на системах с общей памятью накладные расходы оказывают слишком сильное влияние

## 7. Команды для запуска тестов

Команды сборки:

```bash
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake --build build --config Release --parallel
cd build/bin
```

Команды запуска тестов функциональности (SEQ, OMP, TBB, STL, ALL):

```bash
.\ppc_func_tests.exe --gtest_filter="*zagryadskov_m_complex_spmm_ccs_seq_enabled*"
$env:PPC_NUM_THREADS="<количество потоков>"
.\ppc_func_tests.exe --gtest_filter="*zagryadskov_m_complex_spmm_ccs_omp_enabled*"
.\ppc_func_tests.exe --gtest_filter="*zagryadskov_m_complex_spmm_ccs_tbb_enabled*"
.\ppc_func_tests.exe --gtest_filter="*zagryadskov_m_complex_spmm_ccs_stl_enabled*"
mpiexec -n <количество процессов> .\ppc_func_tests.exe \
--gtest_filter="*zagryadskov_m_complex_spmm_ccs_all_enabled*"
```

Команды запуска тестов производительности (SEQ, OMP, TBB, STL, ALL):

```bash
.\ppc_perf_tests.exe --gtest_repeat=5 --test_filter="*task_run_zagryadskov_m_complex_spmm_ccs_seq_enabled"
$env:PPC_NUM_THREADS="<количество потоков>"
.\ppc_perf_tests.exe --gtest_repeat=5 --gtest_filter="*task_run_zagryadskov_m_complex_spmm_ccs_omp_enabled*"
.\ppc_perf_tests.exe --gtest_repeat=5 --gtest_filter="*task_run_zagryadskov_m_complex_spmm_ccs_tbb_enabled*"
.\ppc_perf_tests.exe --gtest_repeat=5 --gtest_filter="*task_run_zagryadskov_m_complex_spmm_ccs_stl_enabled*"
mpiexec -n <количество процессов> .\ppc_func_tests.exe --gtest_repeat=5 \
--gtest_filter="*task_run_zagryadskov_m_complex_spmm_ccs_all_enabled*"
```

## 8. Заключение

Реализованы и протестированы последовательная а также параллельные реализации алгоритма умножения двух
разреженных матриц в формате **CCS** с элементами комплексного типа с использованием технологий
SEQ, OMP, TBB, STL и ALL. В результате экспериментов подтверждена корректность всех реализаций,
вычислено время работы алгоритмов и проведено сравнение с последовательной реализацией. По медианам лучшими
реализациями оказались технологии `OMP` и `STL` с максимальным ускорением `5.73` раз относительно
последовательной реализации. `TBB` также показывает неплохой прирост, но немного меньше `OMP` и `STL`.
Технология `ALL` также показала хорошую масштабируемость и прирост, однако является менее эффективной из-за
накладных расходов на пересылки данных.
