## Введение

В данной работе исследованы несколько реализаций алгоритма умножения двух разреженных матриц в формате
**CCS** (Compressed Column Storage, см. [1]) с элементами комплексного типа `std::complex<double>`.
Рассматривались реализации:

- Без использования технологий параллелизма (последовательная версия, технология **SEQ**);
- С использованием библиотеки OpenMP [2] (технология **OMP**);
- С использованием библиотеки OneTBB [3] (технология **TBB**);
- С использованием стандартной библиотеки `threads` (технология **STL**);
- С использованием MPI [4] для параллелизма на уровне процессов совместно с OpenMP
для параллелизма на уровне потоков (технология **ALL**)

---

## Постановка задачи

Необходимо для двух матриц в формате **CCS** вычислить их произведение и затем сохранить его в формате **CCS**.
Произведение двух матриц A = a[i, j], B = b[j, k] (i = 1, m; j = 1, n; k = 1, p; a[i, j] ∈ **C**; b[i, j] ∈ **C**)
равно матрице C со элементами, выражающимися по формуле: C = c[i, k] = Σ(a[i, j] * b[j, k]), j ∈ 1, n.
Матрицы считаются разреженными, если количество нулевых элементов в ней гораздо больше количества
ненулевых элементов, или (что эквивалентно) если количество ненулевых элементов в ней порядка
одной из размерностей.
В формате **CCS** разреженная матрица m x n с nz - количеством ненулевых элементов представляется в виде 3х массивов:

- Массив `row_ind` размера nz, содержащий координату строки соответствующего элемента;
- Массив `values` размера nz, содержащий значение соотвествующего ненулевого элемента;
- Массив `col_ptr` размера n + 1, содержащий индекс начала нового столбца в массивах `row_ind` и `values`;

В ходе выполнения задачи требуется:

1. Реализовать последовательную версию данного алгоритма;
2. Реализовать параллельные версии с использованием технологий OMP, TBB, STL и ALL
3. Проверить корректность реализаций и провести экспериментальные замеры времени выполнения;
4. Сравнить время работы параллельных реализации с использованием различного количества потоков
и процессов (для ALL) с последовательной реализацией

## Описание алгоритмов

### Технология SEQ

С учетом формата хранения матриц, "наивная" реализация произведения матриц
без использования технологий параллелизма будет такова:

- Левая матрица последовательно умножается на столбцы правой матрицы;
- Для каждого столбца правой матрицы результат умножения сохраняется в результирующую матрицу с
сохранением разреженной структуры;

Ниже приведён псевдокод данного алгоритма:

```cpp
for j ∈ [1; n-1] do:
  row.clear()

  for k_b ∈ [b.col_ptr[j]; b.col_ptr[j+1]] do:
    ind = b.row_ind[k_b]
    for k_a ∈ [a.col_ptr[ind]; a.col_ptr[ind+1]] do:
      row[a.row_ind[k_a]] += b.values[k_b] * a.values[k_a]
    end for
  end for

  for k_c ∈ [1; p] do:
    if row[k_c] not 0.0 do:
      c.values.push_back(row[k_c])
      c.row_ind.push_back(k_c)
      ++c.col_ptr[j+1]
    end if
  end for

  c.col_ptr[j+1] += c.col_ptr[j]
end for
```

Несмотря на кажущуюся простоту такого алгоритма, у него есть 2 существенных недостатка:

1. Во втором внутреннем цикле по `k_c` мы обязаны пробежать все элементы массива `row`,
что может существенно понизить эффектиность алгоритма, так как в этом массиве может быть
много нулевых элементов. Решением этой проблемы будет добавление массива `marker`, который
содержит индексы всех ненулевых элементов в массиве `row`;
2. Во втором внутреннем цикле по `k_c` происходит выделение памяти для новой матрицы c,
а реализация многопоточного выделения памяти в общем случае невозможна.
Чтобы этот алгоритм подходил для параллелизации, нужно вынести выделение памяти из умножения,
образовав **двухфазный алгоритм**;

Первую фазу такого двухфазного алгоритма обычно называют **Символьной фазой (symbolic phase)**,
а вторую - **Численной фазой (numeric phase)**.

**1. Символьная фаза (symbolic phase):**  
Для каждого столбца правой матрицы определяется структура соответствующего столбца результата.

- Выполняется обход ненулевых элементов и формируется множество индексов строк.
Для этого используются локальные для потоков **вектора-индикаторы (marker)**;
- Вычисляется массив **col_ptr** результирующей матрицы;

**2. Численная фаза (numeric phase):**  
Для каждого столбца правой матрицы вычисляется столбец итоговой матрицы.

- Для каждого столбца используются локальные для потоков вектора **acc** аккумулирования результата и
**marker** для исключения повторной обработки столбцов;
- Численный результат записывается в заранее выделенные векторы результирующей матрицы;

Использование двухфазного подхода при параллельных реализацияъ позволяет избежать
гонок данных или блокировок потоков при записи численного результата в итоговую матрицу,
поскольку после выделения памяти на символьной фазе каждый поток
будет заполнять свои столбцы результирующей матрицы независимо от других столбцов.

### Технология OMP

Выполняется умножение по описанному выше двухфазному алгшоритму. При этом обработка столбцов
правой матрицы и вычисление результирующих столбцов левой матрицы выполняется
параллельно и независимо в каждой из фаз.
Для обеспечения параллелизма используется директива препроцессора `omp parallel` из библиотеки **OpenMP**
с разбиением столбцов потокам. При этом каждому потоку назначается функция со своими параметрами индексов столбцов.

### Технология TBB

Выполняется умножение по описанному выше двухфазному алгшоритму. При этом обработка столбцов
правой матрицы и вычисление результирующих столбцов левой матрицы выполняется
параллельно и независимо в каждой из фаз.
Для обеспечения параллелизма используется `tbb::parallel_for` функция из библиотеки **oneTBB**
с разбиением столбцов потокам. Для обеспечения локальности по потокам векторов маркера `marker` и
аккумулятора `acc` используется специализированный контейнер `tbb::enumerable_thread_specific`.

### Технология STL

Выполняется умножение по описанному выше двухфазному алгшоритму. При этом обработка столбцов
правой матрицы и вычисление результирующих столбцов левой матрицы выполняется
параллельно и независимо в каждой из фаз. Каждому потоку назначается функция с аргументами,
соответствующими номеру потока.
Для обеспечения параллелизма используется подход на основе создания контейнера `std::thread`, создания и добавление
в него потоков, которым назначаются соответствующие функции и последующего вызова `.join()`

### Технология ALL

Умножение с использованием параллелизма на уровне процессов при помощи MPI выполняется по алгоритму,
состоящему из нескольких этапов:

**С точки зрения процессов выполняется:**

- Пересылка матрицы **A** всем MPI-процессам с помощью функции `BcastCCS`;
- Распределение столбцов матрицы **B** на корневом процессе, создание локальных для процессов подматриц
**B**, называмых **local_B** и их рассылка процессам с помощью функции `ScatterB`;
- Параллельное умножение **A*local_B = local_C** на каждом процессе. При этом сама операция выполняется
параллельно на **общей памяти** с использованием потоков по **двухфазному алгоритму**, полностью
аналогичному алгоритму, использующемуся задаче для технологии OMP. Алгоритм описан ниже;
- Пересылка **local_C** корневому процессу и сборка этих подматриц в итоговую матрицу **C** с помощью
функции `GatherC`;
- После умножения в функции PostProcessing пайплайна происходит рассылка матрицы **C**
остальным процессам для обеспечения успешного прохождения тестов всеми процессами;

В данном алгоритме параллельно на каждом из процессов выполняется непосредственное умножение
**A*local_B = local_C**. Однако, копирование данных в подматрицы **local_B** и их рассылка,
а также получение подматриц **local_C** и вычисление префиксной суммы `col_ptr` для матрицы **C**
происходит последовательно только на корневом процессе, что может помешать линейному ускорениию
относительно последовательной версии при увеличении количества процессов.

**С точки зрения потоков:**

Выполняется умножение по описанному выше двухфазному алгоритму, используется технология OpenMP.
При этом обработка столбцов матрицы **local_B** и вычисление результирующих столбцов **local_C**
выполняется параллельно и независимо в каждой из фаз. При этом каждый поток
выполняет функции symbolic и numeric, работая со своим диапазоном столбцов от `jstart` до `jend`.

Использование двухфазного подхода позволяет избежать гонок данных или блокировок потоков при записи
численного результата в итоговую матрицу, поскольку после выделения памяти на символьной фазе каждый поток
будет заполнять свои столбцы результирующей матрицы независимо от других столбцов.
Также при таком подходе максимальное количество операций производится параллельно в каждом потоке,
последовательно выполняется только выделение памяти для матрицы **C** и подсчет префиксной суммы
столбцов `col_ptr`.

## Результаты экспериментов и подтверждение корректности

Для **функционального тестирования** были выбраны несколько предпосчитанных примеров, проверяющих
краевые случаи работы алгоритма. Сравнение результатов выполнялось с учетом погрешности операций с
плавающей точкой, которая не должна превышать `1e-14`.

Для **оценки производительности** использовался генератор разреженных ленточных матриц 40000 на 40000,
ширина ленты 100, ненулевые значения в диапазоне от 1.0 до 2.0 для действительной и мнимой компонент.
Результаты тестирования представлены ниже:

| Версия алгоритма | 2 потока (1+2; 2+1) | 3 потока (-) | 6 потоков (2+3) | 12 потоков (2+6) |
| ---------------: | -------: | -------: | --------: | ---------: |
| Последовательная | 0.785 | 0.785 | 0.785 | 0.785 |
| Параллельная (OpenMP) | 0.436 | 0.290 | 0.174 | 0.110 |
| Параллельная (OneTBB) | 0.413 | 0.277 | 0.154 | 0.091 |
| Параллельная (STL) | 0.456 | 0.310 | 0.196 | 0.149 |
| Параллельная (MPI + OpenMP) | 0.523; 0.656 | - | 0.243 | 0.209 |

| Версия алгоритма | 2 потока (1+2; 2+1) | 3 потока (-) | 6 потоков (2+3) | 12 потоков (2+6) |
| ---------------: | -------: | -------: | --------: | ---------: |
| Ускорение OMP (раз) | 1.8 | 2.7 | 4.5 | 7.1 |
| Ускорение TBB (раз) | 1.9 | 2.8 | 5.1 | 8.6 |
| Ускорение STL (раз) | 1.7 | 2.5 | 4.0 | 5.2 |
| Ускорение ALL (раз) | 1.5 | 1.2 | 3.2 | 3.8 |

В приведённых таблице во втором, третьем и последующих столбцах представлено время выполнения в секундах
при использовании 2, 3, 6, и 12 потоков. В скобках указана конфигурация, которая использовалась для запуска
технологии ALL: количество процессов + количество потоков. Тестирование производилось на
6-ядерном процессоре Intel i5-12400f.

## Выводы из результатов

Параллельная реализация до **8.6 раз**. Это демонстрирует, что алгоритм хорошо приспособлен к параллелизму.
Наибольшее ускорение было достигнуто при использовании технологии OneTBB. Это объясняется тем, что
именно в этой библиотеке благодаря планировщику задачи распределяются потокам наиболее эффективным
образом. Наибольшая эффективность для всех технологий достигается при использовании 2 потоков.
Также предсказуемым результатом является то, что для системы на общей памяти (которой является
устройство, на котором проводилось тестирование) технология ALL отработала хуже остальных технологий.
Использовать такой подход имеет смысл только для многопроцессорных систем с распределённой памятью.
